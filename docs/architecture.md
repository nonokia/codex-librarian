# Codex Librarian — アーキテクチャ設計書 (v0.1)

> コードベースの「理解負債」を返済するAIレビュー&知識資産化プラットフォーム。
> NDC26で発表されたNexon CodeOnBoardのコンセプトを参照しつつ、個人開発スケールに最適化して再設計する。
> 本書は「なぜ・何を・どこで割り切るか」を定義する。実装詳細(HOW)は本書を入力として別途設計する。

---

## 1. 問題定義 — なぜ作るか

生成AIの普及でコードを「書く」コストは劇的に下がったが、「理解する・検証する」コストは人間に残ったままである。結果として、開発組織のボトルネックはコード生成からコード理解へ移動した。これが理解負債(Comprehension Debt)であり、以下の形で顕在化する。

1. **レビューの文脈欠如:** 既存のAIコードレビューはPRのdiffしか見ない。変更された関数の呼び出し元・呼び出し先を知らないため、「diff単体では正しいが、システム全体では壊れている」変更を見逃す。
2. **知識の非構造化:** コードベースは「本が無秩序に積まれた倉庫」であり、開発者は理解のたびにゼロから探索する。理解の成果が個人の頭の中にしか残らず、資産化されない。
3. **AIレビューの検索精度:** LLMに文脈を渡すための検索(retrieval)自体の精度が低いと、レビュー品質は頭打ちになる。Nexonの発表でも初期検索成功率は43%に留まり、自己改善ループの導入で平均マッチ率92%まで引き上げたことが成果の中核だった。

**本プロダクトの目的:** diffではなくコードグラフを一次情報源とし、変更の影響範囲を含む文脈をLLMに供給することで、レビュー品質と検索精度を継続的に改善するシステムを作る。

## 2. プロダクト原則 — 判断に迷ったらここに戻る

- **Graph-first, not diff-first.** レビューの単位はdiffではなく「diffが触れたグラフ上の近傍」である。すべての機能はコードグラフを前提に設計する。
- **Retrievalは計測可能でなければならない.** 検索精度を数値化する評価ハーネスを機能実装より先に作る。Nexonが43%→92%を語れたのは計測基盤があったからであり、これがないと自己改善ループは作れない。
- **ローカル完結・単一プロセス.** コードを外部に送らない前提(Nexonの内製動機と同じ)。LLM API呼び出し以外はすべてローカルで完結し、運用対象のミドルウェアを持たない。
- **蓄積は副産物として発生させる.** 知識資産化のために開発者へ追加作業を要求しない。レビューと検索の営みの中から自動的にナレッジが蓄積される設計にする。

## 3. スコープ — 何を作り、何を作らないか

### 3.1 スコープ (v1)

- 対象VCS: **Gitのみ**。対象ホスティング: GitHub。
- 対象言語: **TypeScriptのみ**(第一言語)。多言語対応は抽象インターフェースだけ用意し実装しない。
- 対象リポジトリ規模: 〜数十万行。複数リポジトリを同一 SQLite にインデックスして
  横断検索・グラフ探索できる(マルチレポ、#11。schema v2 で `repos` 次元を追加)。
  リポジトリ間の import は抽出だけでは解決不能なので `resolved = 0` のまま隔離する
  (§8 リスク 2 の不変条件を維持)。**欠けているのは「package 名 → repo」という 1 つの
  事実だけ**であり、これを明示宣言(`links.json`)として与えると `librarian link` が
  未解決エッジを再解決する(ADR-8、#27。宣言が無ければ何も起きない — 隔離が既定)。
- 提供機能: (a) PR単位の文脈付きAIレビュー、(b) コードベースへの意味検索+グラフ探索Q&A、(c) 検索パターンの自己改善ループ。

### 3.2 Non-goals — Nexonとの意図的な差分

Nexonの自作動機はPerforce/SVN対応・独自スクリプト言語・社内セキュリティ要件だった。本プロダクトは個人開発・ポートフォリオ目的であるため、以下を明示的に捨てる。

- Perforce/SVN対応、ゲームエンジン固有言語対応
- マルチテナント、組織権限管理
- モデルのファインチューニング(LoRA等)。retrieval改善で戦う
- IDEプラグイン(v1ではCLI + GitHub + Webの3面で十分)

## 4. システム分解

システムは4つのコンポーネントと1つの横断ループで構成する。

```
                 ┌──────────────────────────────┐
   git repo ───▶│ ① Indexer                     │
                 │  (抽出・解決・埋め込み)        │
                 └──────────────┬───────────────┘
                                ▼
                 ┌──────────────────────────────┐
                 │ ② Knowledge Store (SQLite)    │
                 │  symbols / edges / vectors    │
                 └──────────────┬───────────────┘
                                ▼
   PR diff ────▶ ┌──────────────────────────────┐      ┌────────────┐
   質問 ───────▶│ ③ Context Engine              │◀────▶│ ⑤ Retrieval │
                 │  (グラフ展開+意味検索→文脈束) │      │  Cache/自己 │
                 └──────────────┬───────────────┘      │  改善ループ │
                                ▼                       └────────────┘
                 ┌──────────────────────────────┐
                 │ ④ Interfaces                  │
                 │  CLI / GitHub PRコメント / Web │
                 └──────────────────────────────┘
```

### ① Indexer — コードを構造化する

リポジトリを走査し、シンボル(関数・クラス・型)、その定義位置、シグネチャ、docstring、そしてシンボル間のエッジ(calls / imports / extends / references)を抽出する。同時に各シンボルの要約テキストを埋め込みベクトル化する。

設計上の要点は「抽出器(Extractor)を言語ごとに差し替え可能なインターフェースにする」こと。v1実装はTypeScript Compiler APIによる**型解決済み**の抽出とする(理由は§5-ADR2)。

インクリメンタル性が必須要件。コミットごとのフルインデックスは許容できないため、変更ファイル+その参照元のみ再解析する。ファイルハッシュとシンボルIDの安定化戦略はHOW側の主要設計課題として委譲する。

### ② Knowledge Store — 単一SQLiteに3つの役割を同居させる

Nexonは RDB / Vector DB / Knowledge Graph の3ストア構成だが、これは組織スケールの要件。本プロダクトは**SQLite単一ファイル**に集約する(理由は§5-ADR1)。

- RDB役: symbols, files, spans テーブル
- Graph役: edges テーブル(from_symbol, to_symbol, kind)。k-hop探索は再帰CTEで実現
- Vector役: sqlite-vec 拡張による埋め込み格納・近傍検索

概念データモデル(詳細スキーマはHOWに委譲):

| エンティティ | 主な属性 |
|---|---|
| Symbol | id(安定ハッシュ), kind, name, file, span, signature, summary |
| Edge | from, to, kind(calls/imports/extends/references) |
| Embedding | symbol_id, vector |
| RetrievalLog | query, strategy, hits, outcome(採用/不採用), latency |
| PatternCache | diffシグネチャ→有効だった探索戦略 |

### ③ Context Engine — 本プロダクトの心臓部

入力(PR diffまたは自然言語の質問)から、LLMに渡す「文脈束(Context Pack)」を組み立てる。競合との差別化はすべてここに宿る。

パイプラインは2段構成:

1. **決定的展開(deterministic):** diffが触れたシンボルを起点に、callers/calleesをk-hop展開。トークン予算内で「変更点に近い順・エッジ種別の重み順」に貪欲に詰める。ここはLLMを使わない。再現性と速度を担保する層。
2. **意味的補完(semantic):** 予算が余る場合、またはdiffから意図が読み取れる場合に、埋め込み検索で「構造的には遠いが意味的に近い」コード(類似実装、関連テスト、規約ドキュメント)を補う。

Context Packは構造化フォーマット(変更コード / 呼び出し元 / 呼び出し先 / 類似実装 / 関連テスト、の区画付き)としてLLMに渡す。レビュー生成そのものはLLM(Claude API)の仕事であり、本システムの価値は**何を渡すか**に集中させる。

### ④ Interfaces

- **CLI:** `librarian index`, `librarian review <diff>`, `librarian ask "<質問>"`。すべての機能はまずCLIとして存在する(テスト容易性・CI組み込み)。
- **GitHub連携:** GitHub Actions上でCLIを実行し、レビュー結果をPRコメントとして投稿。Webhookサーバは持たない(運用ゼロ原則)。
- **Web UI (Next.js):** コードグラフの可視化と検索Q&A。「倉庫を図書館にする」体験の見せ場であり、ポートフォリオ上の顔になる。ただしフェーズ3まで着手しない。

### ⑤ Self-improving Retrieval Loop — 差別化の核

すべての検索・展開の試行をRetrievalLogに記録し、成果(LLMがその文脈を引用したか、レビュー指摘が人間に採用されたか)をフィードバックする。有効だった「diffの型 → 探索戦略」の対応をPatternCacheに蓄積し、次回以降の同型diffで再利用する。

v1のフィードバック信号は簡素でよい: (a) LLMレスポンスがContext Pack内のどの区画を引用したかの自動判定、(b) PRコメントへの👍/👎リアクション。凝った強化学習は不要で、「成功パターンのキャッシュと優先適用」だけでNexonは43%→92%を達成している。ここが本プロダクトの技術的な主張点になる。

## 5. 主要アーキテクチャ判断 (ADR要約)

**ADR-1: ストアはSQLite単一ファイル(Neo4j/Qdrant/Postgresを採用しない)**
理由: 個人開発で3ミドルウェアの運用は開発速度を殺す。数十万行規模ならedgesテーブル+再帰CTEでk-hop探索は十分速く、sqlite-vecで近傍検索も足りる。「コードを外に出さない・単一バイナリで動く」というプロダクト原則とも一致。スケール限界に達したときの移行パス(edgesのエクスポート形式)だけ定義しておく。
トレードオフ: 巨大モノレポ・高度なグラフアルゴリズムは諦める。それはNexonの土俵であり、本プロダクトの土俵ではない。

**ADR-2: v1抽出器はtree-sitterではなくTypeScript Compiler APIを使う**
理由: call graphの価値は参照解決の精度で決まる。tree-sitterは構文しか見えず、動的importやメソッド解決で偽エッジ/欠落エッジが多発し、「グラフを一次情報源にする」という原則が崩れる。対象をTSに絞った以上、型情報込みの正確なグラフを取る。
トレードオフ: 多言語展開のコストが上がる。Extractorインターフェースで吸収し、多言語時にtree-sitter系(またはSCIP)実装を追加する。

**ADR-3: Context Engineは「決定的展開が主、LLMエージェント探索は従」**
理由: エージェントに毎回自由探索させる方式はコスト・レイテンシ・再現性のすべてで劣化し、精度の計測も困難になる。決定的パイプラインを基盤にし、PatternCacheにない未知パターンに遭遇したときだけエージェント探索にフォールバック→成功したらキャッシュに昇格、という構造にする。これが自己改善ループの実体でもある。

**ADR-4: 評価ハーネスを最初のマイルストーンにする**
理由: 「retrieval精度n%」を語れないプロダクトは、この領域では既存AIレビューとの差別化を示せない。対象リポジトリ(著名OSSを1つ選定)の過去PRから「このdiffのレビューに必要だった文脈」の正解セットを数十件作り、match率を自動計測できる状態を先に作る。以降のすべての変更はこの数値で正当化する。

**ADR-5: 言語はTypeScriptで統一、モノレポ構成**
理由: 開発者(自分)の主戦場であり、Indexer(Compiler API)・CLI・Next.js UIまで単一言語で貫ける。Loreのvisual diff UIプロジェクトと技術スタックが揃い、ポートフォリオとして一貫する。

**ADR-6: 抽出器と store の間の交換フォーマットは SCIP ベース層 + 拡張層の二層(SCIP+)**
理由: (1) 標準フォーマットの取り込み口により、native 抽出器を書かずに外部 `scip-*` インデクサの言語を追加できる。(2) index を `.scip` として export でき、SCIP エコシステムと相互運用できる。(3) retrieval を駆動する固有信号(エッジ種別・unresolved・testblock)は SCIP に表現が存在しないことを仕様確認で確定済みのため、サイドカー拡張(ext)を正とし、ベース SCIP は標準準拠の投影とする。ADR-1 とは非衝突(SCIP+ は抽出→store 間の契約で、保存先は SQLite のまま)。ADR-2 とも非衝突(抽出は言語ごと native のまま。#7/#8 で却下したのは「外部 SCIP インデクサへの置き換え」であり、本 ADR は交換フォーマットの標準化)。
トレードオフ: 参照位置の追加取得と二層の維持コスト。native 経路の精度は ext が担うため SCIP 化そのものによる精度向上は無い(相互運用と言語追加コスト削減への投資)。外部 `.scip` はエッジ種別・unresolved・testblock ネストが欠落した degrade 動作となり、その実測値を `docs/scip-baseline.md` に記録する。設計の全文は `docs/scip-design.md`(issue #16)。

**ADR-7: 抽出器はプロトコル準拠のプラグイン。発見・登録は明示レジストリ、ワイヤ契約は SCIP+ 封筒。**
理由: librarian のコア価値は store/retrieval であり、パース・抽出は第三者が差し替え・追加できるべき。ADR-6 の SCIP+ 封筒(stdin/stdout の単一 JSON、protobuf 非依存)+ moniker→id の決定的再計算(#16)+ repo-unaware invariant(#11)で、事実上のプラグイン ABI は既に存在する。ADR-7 はこれを公開契約に格上げし、(1) 発見・登録を `.librarian/extractors.json`(拡張子→コマンドの明示宣言)に集約、(2) 言語別アダプタを 1 つの汎用サブプロセスランナー(`SubprocessExtractor`)に畳み、(3) `--capabilities` ハンドシェイクで封筒バージョンを交渉する。信頼モデルは**明示登録のみ・自動ダウンロード無し・PATH 規約による暗黙発見を採らない**(暗黙の信頼と環境差の非決定性を排除)。ADR-2 と非衝突(抽出は言語ごと native のまま。プラグイン化は「発見・登録の外部化」であって「TS Compiler API を捨てる」ことではない — TS は in-process 実装のまま特別扱い)。ADR-6 と非衝突(ワイヤ契約は SCIP+ 封筒そのもの。プロトコルはその発見・実行規約を足すだけ)。ADR-1 と非衝突(保存先は SQLite のまま)。
トレードオフ: 公開契約はバージョニングとの後方互換コミットメント(封筒スキーマは追加のみ、major は `--capabilities` で交渉)。プラグイン = 任意コマンド実行の信頼境界を利用者に開く。既存 Go/PHP はレジストリ経由で動き eval 完全一致(#16 と同じゲート)。設計の全文は `docs/plugin-protocol.md`(issue #22)。

**ADR-8: リポジトリ間 import 解決は index の一部ではなく、明示宣言を入力とする後段の別ステップ(`librarian link`)。**
理由: 「`@acme/core` は隣の repo だ」はどちらのツリーにも書かれていない外部知識であり、抽出器が知り得ない(repo-unaware invariant、#11)。これを index に埋め込むと index の入力が「その repo のツリー」から「db 全体の状態」に広がり、決定性(同じツリー → 同じ行)が壊れる。よって (1) 写像は `.librarian/links.json` の**明示宣言のみ**(package.json 等からの自動推測はしない — ADR-7 と同じ信頼モデル)、(2) call site の名前は**推測で一致させず**、抽出器が記録した import binding エッジ(`<spec>#<imported>`、`docs/plugin-protocol.md`)を辿って束縛する。曖昧(対象 repo に同名の module-scope 宣言が複数)なら**リンクせず拒否**、メソッド・default/namespace import は `resolved = 0` のまま残す。(3) `link` は冪等かつ可逆(`--clear` で抽出直後の行に戻る)。宣言が無い db では cross-repo エッジは 1 本も作られず、graph/pack/eval は #27 以前と同一。
トレードオフ: 完全性は捨てている(メソッド越え・binding を吐かない言語は繋がらない)が、それが §8 リスク 2 の「偽エッジを作らない」を守る対価。宣言の運用コスト(repo を跨ぐ度に 1 行)と、`index` 後に `link` を再実行する運用(変更ファイルの cross-repo エッジは再抽出で unresolved に戻る)を利用者に負わせる。数値は `docs/cross-repo-baseline.md`(同一 golden で link なし 0.429 → link あり 1.000)。

**ADR-9: フレームワーク規約による動的ディスパッチは新エッジ種別 `dispatches` としてグラフに乗せる。抽出器は `resolved=false` の事実だけを記録し、解決は規約に基づく後段の別ステップ(`librarian resolve-dispatches`)。**
理由: 「呼び出し関係が識別子ではなく文字列/設定/命名規約で表現され、実行時にフレームワークが解決する」パターン(CakePHP の `redirect(['controller'=>..,'action'=>..])`/`setAction`、Next.js のファイルルーティング、Django/Flask のルーティング等)は言語横断で共通のクラスであり、汎用パーサ(nikic/php-parser 等)は関知しない。結果として画面遷移フロー(アクションA→B→C)がグラフに存在せず、`ask`/`graph`/retrieval から見えない(発端は #43)。個々の抽出器にフレームワーク固有の特殊ケースを積むと発散し ADR-2/7 のプラグイン契約を汚す(却下案 A)。プロンプト側でコントローラー全文を混ぜるヒューリスティックはグラフに乗らず PatternCache 等から不可視で ADR-3 の決定性優先に反する(却下案 B)。よって (1) `dispatches` を正式なエッジ種別として定義し、(2) 抽出器は検出した遷移を **`resolved=false`・`dispatch <controller>#<action>`** の形で記録するだけ(ADR-8 の import binding と同型:プラグインは事実だけ、解決は明示的な後段)、(3) `resolve-dispatches` が CakePHP の命名規約(`<name>Controller` + 同名 public メソッド)で**推測ではなく規約に基づき**束縛する。曖昧(同名 controller が複数ファイル)なら**繋がず拒否**、規約対象が存在しなければ `resolved=0` のまま残す。冪等かつ可逆(`--clear`)。リテラル文字列でないルーティング(変数・式)は原理的に静的解決不可能なので**そもそも記録しない**(スコープ外)。ADR-2/7 と非衝突(`dispatches` はプラグインの任意 opt-in、既存 call graph 精度に影響しない)。ADR-8 と非衝突だが別コマンド(`link` は cross-repo の明示宣言写像、`resolve-dispatches` は intra-repo の命名規約 — 入力も信頼モデルも別)。
トレードオフ: `link` と同じく完全性は捨てる(継承先のアクション、public 可視性の厳密判定、他フレームワークは未対応 — Step 2 以降の課題)が §8 リスク 2 の「偽エッジを作らない」を優先。`dispatches` の retrieval 重みは暫定 0.85(calls=1.0 と references=0.7 の中間 — 遷移は実際の制御フローだが呼び出しよりは弱い)で、最適値は ADR-4 の掃引に委ねる。数値は `docs/dispatch-baseline.md`(同一 golden で resolve なし 0.273 → あり 1.000)。

**ADR-10: LLM 呼び出しはプロバイダ抽象(`src/llm/`)経由。プロバイダは env による明示選択のみ、既定は Anthropic 公式 API、暗黙フォールバックなし。**
理由: LLM 呼び出しが `review`(CLI)と `ask`(Web)の 2 箇所で SDK 直依存になっており、Anthropic 公式 API に直接到達できない環境(プロキシ/ゲートウェイ経由必須の組織)では個別パッチが要る(発端は #42)。個々の環境向けパッチはコードベースに環境差分を漏らすので却下。よって (1) プロダクト本体(`src/core`/`src/app`/`web/app`)は「system prompt + user content (+ 構造化出力スキーマ) → テキスト or 構造化オブジェクト」という抽象(`LlmProvider`)のみ扱い、プロバイダ名(SDK・env 名・型名)を知らない。(2) 選択は `LLM_PROVIDER` env の明示指定のみ(`anthropic` / `openai-compatible`)。未知の値はエラーで、あるプロバイダの失敗が別プロバイダへ暗黙に切り替わることはない — ADR-7 の抽出器レジストリと同じ信頼モデル。(3) 既定プロバイダは Anthropic 公式 API で、`ANTHROPIC_API_KEY` だけで動く従来の挙動は不変。(4) 構造化出力は最大公約数で抽象化する: ネイティブのスキーマ強制(Anthropic の output_config)があれば使い、無いプロバイダ(`openai-compatible`)は JSON Schema を system prompt に埋め込み応答をパースする(フェンス許容)。パース失敗は自動リトライせず型付きエラー(`LlmParseError`)で決定的に表面化させる(ADR-3 の決定性優先と同じ理由)。(5) 特定サードパーティのベンダー名・エンドポイントはコード/公開ドキュメントに持ち込まない — `openai-compatible` は base URL・鍵・モデル名を env(`LLM_OPENAI_COMPATIBLE_BASE_URL` / `LLM_OPENAI_COMPATIBLE_API_KEY` / `LLM_MODEL`)で受け取る汎用実装。(6) モデル指定は `LLM_MODEL` に統一し、既存の `LIBRARIAN_MODEL` は後方互換エイリアスとして残す(優先度: `--model` > `LLM_MODEL` > `LIBRARIAN_MODEL` > プロバイダ既定)。第三者プロバイダの外部レジストリ化(ADR-7 型の `.librarian/llm.json`)は YAGNI として見送り — ビルトイン 2 実装で要件を満たし、必要になったら ADR 変更で拡張する。ADR-5 と非衝突(web は親 dist/ の registry を import するだけで、ロジックの再実装はしない)。レイヤ DAG 上では `src/llm/` は他レイヤに依存しない葉で、`app/` と `web/` が依存する(`core/` は従来どおり LLM に触らない)。

## 6. フェーズ計画

各フェーズは「単体でデモ可能な成果物」を持つ。フェーズ内の実装設計はOpus/Sonnetに委譲する単位でもある。

- **Phase 0 — 計測基盤 (最優先):** 対象OSSリポジトリ選定、正解セット作成、retrieval match率の自動評価ハーネス。成果物: 評価レポートを出すCLI。
- **Phase 1 — Indexer + Store:** TS Compiler APIでsymbols/edges抽出、SQLiteスキーマ、インクリメンタル更新、埋め込み生成。成果物: `librarian index` と統計ダッシュボード(CLI)。
- **Phase 2 — Context Engine + Review:** 決定的展開、Context Pack組み立て、Claude APIでのレビュー生成、GitHub Actions連携。成果物: 実PRへの文脈付きレビューコメント。**ここがMVP**。
- **Phase 3 — Web UI:** グラフ可視化・意味検索Q&A(Next.js)。成果物: 「図書館」体験のデモ。
- **Phase 4 — Self-improving Loop:** RetrievalLog、フィードバック信号、PatternCache、エージェント探索フォールバック。成果物: 精度改善の時系列グラフ(43%→92%に相当する自分の数字)。

## 7. 成功指標

- **Retrieval match率:** 正解セットに対する文脈一致率。Phase 2終了時60%以上、Phase 4で85%以上を目標。
- **レビュー有効率:** 生成された指摘のうち「diff外の文脈を根拠にした指摘」の割合。diffだけ見るレビューとのA/B比較で優位を示す。
- **コスト/レイテンシ:** PRあたりのトークン消費とレビュー完了時間。決定的展開比率が上がるほど下がるはずで、自己改善の副次指標になる。

## 8. 主要リスク

1. **正解セット作成の工数:** Phase 0が退屈で挫折ポイントになりうる。→ 30件の小さな正解セットで開始し、以降は運用中のログから半自動で追加する。
2. **TS Compiler APIの複雑性:** 型解決の網羅は沼。→ 「解決できたエッジのみ格納、未解決は unresolved フラグ付きで保持」とし、完全性より計測可能性を優先。
3. **グラフ展開のトークン爆発:** hop数を上げると文脈が予算超過する。→ トークン予算をContext Engineの第一級制約とし、詰め込み順序(エッジ重み)自体をPatternCacheの学習対象にする。

## 9. HOW委譲時の指示テンプレート

Opus/Sonnetへ各フェーズを委譲する際は、本書全体+該当フェーズ番号を渡し、以下を要求すること: (1) 本書の原則・ADRに反する設計をしない、(2) 反する必要が生じた場合は実装せずADR変更提案として返す、(3) 成果物は必ずPhase 0の評価ハーネスで計測して数値を添える。
